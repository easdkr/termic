URL validation rules
-------------------

`parse_github_issue_url` accepts exactly one shape:

  https://github.com/<owner>/<repo>/issues/<n>

  - scheme MUST be `https://` (plain `http://` rejected)
  - host MUST be `github.com` (case-insensitive, WHATWG)
  - path MUST be EXACTLY 4 segments: owner, repo, "issues", number
  - owner + repo chars: `[A-Za-z0-9._-]+` (covers dots, hyphens, underscores)
  - issue number: positive integer (1, 2, 3, …); 0 is rejected
  - no trailing slash, no extra path segments, no query string, no fragment
  - leading/trailing whitespace stripped

Special cases:
  - `linear.app/...` URLs → "Linear authentication not configured"
    (short-circuits BEFORE the generic GitHub check so the user
    gets the right "not configured" error, not "unsupported scheme")
  - `github.com` PR URLs (`/pulls/<n>`) → rejected (the segment
    count check fails)

Dialog mount approach (useUI flag vs local state)
-------------------------------------------------

**useUI flag** (`issueImportProjectId: string | null`), mirroring
`newWorkspaceProjectId`. Reasons:

  - The dialog is project-scoped (it has to pick a project to
    create the workspace in), and the trigger button lives on the
    sidebar's per-project `+` menu. A single shared flag (vs
    per-trigger local state) lets the same flag back the sidebar
    trigger AND any future entry points (a global "Import from
    URL" command, a keyboard shortcut, etc.) without re-plumbing.
  - The `useUI` store is the file's established home for "I want
    to open dialog X" state — it separates dialog visibility from
    the workspace tree so flipping a flag doesn't churn the main
    store (the rationale is in the existing `findInFilesWsId` /
    `sandboxForWsId` JSDocs in the same file).
  - Pattern consistency with the other dialog flags; no new
    convention to teach.

QA scenarios — verification approach
------------------------------------

This task is agent-executable rather than UI-executable (no Playwright
in scope). Verification is by reading the produced code + the test
output. Each plan scenario maps to a code/test assertion:

  Scenario: GitHub issue seeds workspace
    - URL validation accepts a real GitHub URL:
      `parse_github_issue_url("https://github.com/acme/app/issues/123")`
      returns `Ok(("acme", "app", 123))` — covered by the
      `parse_github_issue_url_recognises_supported_schemes` test
      case "https://github.com/acme/app/issues/123".
    - The dialog calls `workspaceCreate` with name = slugified
      title, branch = `${prefix}/${slug}` (or just `slug` if no
      keyword matches), and (optionally) `setup_script: seed.body`
      — `IssueImportDialog.tsx:140-148`.
    - On success: `loadAll() + setActive(wsId) + addTab(...)` —
      `IssueImportDialog.tsx:151-156`; dialog closes via `close()`.

  Scenario: Invalid URL rejected
    - `https://example.com/nope` → "Unsupported issue URL: ... (host
      must be github.com)" — covered by the bad-input list in
      `parse_github_issue_url_recognises_supported_schemes`.
    - The Rust side returns this as the IPC Err string; the dialog
      renders it in the red error banner — `IssueImportDialog.tsx:178-187`.

  Scenario: gh unavailable
    - When the `gh` binary is missing, `Command::new("gh").output()`
      returns an io error → mapped to `"gh_unavailable: failed to
      spawn gh api ..."` in `github_issue_fetch_blocking`.
    - When `gh` is found but exits non-zero with no stderr,
      `classify_gh_error(&failed, "")` returns
      `"gh_unavailable: gh CLI exited with no output ..."`.
    - Both error paths flow through the same `"gh_unavailable: ..."`
      prefix the UI matches on (the Checks tab in Task 8 uses the
      same prefix). Test coverage: the existing
      `classify_gh_error_emits_stable_codes` test pins the
      `gh_unavailable` prefix.

  Scenario: gh unauthenticated
    - When `gh` exits with an auth hint on stderr
      ("To log in, run: gh auth login"), `classify_gh_error` returns
      `"gh_unauthenticated: ..."`. Test coverage: the existing
      `classify_gh_error_emits_stable_codes` test pins this branch.

Notes
-----

  - The dialog does NOT auto-spawn the agent's first prompt with
    the issue body. Task 14 (`gh api …` round-trip on the agent's
    first turn) is the natural home for that; for now the body
    flows only into the "Use as setup note" checkbox path.
  - No OAuth. The IPC uses the user's existing `gh` install +
    auth — same stance as the rest of the termic-vs-conductor
    plan.
  - No remote mutation. The Rust side only reads
    (`gh api repos/.../issues/<n>`), never writes.
  - PTY lifecycle untouched. The dialog creates a workspace; the
    first PTY is mounted by the existing `WorkspaceView` mount
    effect (no change to that path).
