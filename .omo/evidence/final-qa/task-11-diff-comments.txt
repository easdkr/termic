SCENARIO 2 — comments persist across remount
---------------------------------------------
Store shape: diffComments: Record<wsId, DiffInlineComment[]> at
src/store/app.ts:93. Workspace-scoped so different workspaces are
isolated. The action mutators (add/update/delete/markPosted) are
immutable — every set() call returns a fresh nextMap with the targeted
wsId entry replaced.

The store is in-memory only (no disk persistence per Task 11 spec —
Task 14 round-trips posted comments through gh api and stamps
remote_id). Within a session:

  1. User adds a comment → store holds it under diffComments[wsId].
  2. User switches to another tab → DiffPane unmounts, MergeView
     destroyed, gutter markers gone.
  3. User switches back → DiffPane remounts → useDiffComments(wsId)
     selector returns the same array (the store didn't change) → the
     line-with-comments Sets rebuild from the same inputs → marker
     DOM re-renders with hasComment=true for the saved line.

The empty constant EMPTY_DIFF_COMMENTS (src/store/app.ts:994) is the
frozen fallback when a wsId has no entry, so the selector stays
referentially stable across renders. This avoids the React 19
"getSnapshot should be cached" warning the plan calls out.

VERDICT: PASS — empty body is rejected at BOTH the store and popover
layers with identical error text, and the persistence is a straight
consequence of the in-memory store + frozen-empty selector + non-unmount
on tab switch (CLAUDE.md "Keep terminals mounted across switches" applies
to the parent WorkspaceView; DiffPane itself does remount but its
selector re-fetches the same array, so the markers come back).
