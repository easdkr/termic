QA Scenario 2: Invalid link name rejected
-----------------------------------------

Mirrors the Rust `validate_link_name` rejection set:

  name='../secrets'  -> REJECTED: invalid link name (no / allowed)
  name='a/b'         -> REJECTED: invalid link name (no / allowed)
  name='a..b'        -> REJECTED: invalid link name (no .. allowed)
  name='.hidden'     -> REJECTED: invalid link name (cannot start with .)
  name=''            -> REJECTED: link name cannot be empty

Note: '../secrets' is rejected by the `/` check before the `..` check
runs (since `..` contains `/`); either rule alone is sufficient. The
`..` rejection is needed for the `a..b` shape (no `/` present).


Behavior summary
----------------

- `materialize_external_dir_links` is called from both
  `workspace_create_sync` (after `files_to_copy` copy, before
  `port` allocation) and `workspace_import_worktree` (after
  `save_workspace`, before `Ok(ws)`). New + imported workspaces
  get the symlinks for free.
- Skip-don't-clobber: a real file at the link path is left alone
  with a logged warning, matching the plan's "do not overwrite real
  files" rule. Verified by `materialize_skips_when_real_file_occupies_link_path`.
- Repair button: `workspace_repair_links` (async + spawn_blocking)
  re-materializes the project's links in EVERY active workspace,
  returns a flat list of result lines for the UI toast.
- Validation: empty / `/` / `..` / leading-dot are all rejected on
  the Rust side. The TS form has a client-side mirror of the same
  rules for fast feedback; the Rust side is the authoritative check.
- IPC contract: snake_case on both sides (project_id, name,
  target_path, workspace_id, etc.) — matches the established
  Task 1 / Task 6 convention.
- No new dependencies; `openDialog({ directory: true })` reuses
  the existing `tauri-plugin-dialog` integration.
